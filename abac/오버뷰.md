# 프로젝트 오버뷰 — 코드 읽는 지도

> 각 파일이 **무엇인지 / 왜 있는지 / 어떻게 연결되는지**를 한눈에 훑기 위한 문서.
> 권한 매트릭스·실행 방법은 [CLAUDE.md](CLAUDE.md) 참조.
> 프론트엔드 쪽 지도는 [`../abac-frontend/README.md`](../abac-frontend/README.md). 백엔드 ↔ 프론트 연결은 [SecurityConfig.java](src/main/java/com/example/abac/config/SecurityConfig.java)의 `corsConfigurationSource()`가 `localhost:5173/5174`를 허용하는 지점.

---

## 읽는 순서 (추천)

코드를 처음 본다면 위에서 아래로 따라가면 한 요청의 생명주기를 자연스럽게 이해할 수 있다.

1. [AbacApplication.java](src/main/java/com/example/abac/AbacApplication.java) — 메인, 전역 어노테이션 2개
2. [SecurityConfig.java](src/main/java/com/example/abac/config/SecurityConfig.java) — 필터 체인 구성 (요청이 어떻게 들어오는가)
3. [JwtAuthenticationFilter.java](src/main/java/com/example/abac/security/jwt/JwtAuthenticationFilter.java) — 토큰을 Principal로
4. [CustomUserPrincipal.java](src/main/java/com/example/abac/security/CustomUserPrincipal.java) — ABAC 속성의 출발점
5. [ExpenseController.java](src/main/java/com/example/abac/domain/expense/ExpenseController.java) — `@PreAuthorize`가 모여있는 곳
6. [ExpensePolicy.java](src/main/java/com/example/abac/policy/ExpensePolicy.java) — 액션 권한 규칙 (ABAC 게이트)
7. [ExpenseSpecifications.java](src/main/java/com/example/abac/domain/expense/ExpenseSpecifications.java) — 데이터 스코프 규칙
8. [ExpenseService.java](src/main/java/com/example/abac/domain/expense/ExpenseService.java) — 상태 전이와 트랜잭션

## 패키지 구조 요약

```
com.example.abac
├── AbacApplication          (부팅 + 전역 스위치)
│
├── common/                  횡단 관심사 — 엔티티 공통 필드, 예외 응답
├── config/                  빈 구성 — 시큐리티, 시딩
│
├── security/                인증 계층 — JWT, Principal, 로그인 API
│   ├── jwt/                 토큰 발급·파싱·필터
│   └── dto/                 로그인 요청·응답
│
├── domain/
│   ├── user/                User 애그리거트 + Role enum
│   ├── department/          Department 애그리거트 (조직 단위)
│   └── expense/             Expense 애그리거트 — 본 프로젝트의 중심 도메인
│       └── dto/             요청·응답 DTO
│
└── policy/                  ABAC 정책 Bean — SpEL에서 호출되는 단일 진실 소스
```

### `common/` — 횡단 관심사
| 파일 | 역할 |
|---|---|
| [BaseTimeEntity.java](src/main/java/com/example/abac/common/BaseTimeEntity.java) | `createdAt`/`updatedAt` 자동 채움. `User`와 `Expense`가 상속. `AuditingEntityListener` 연결. |
| [ApiError.java](src/main/java/com/example/abac/common/ApiError.java) | 에러 응답 포맷 record. status / error / message / timestamp. |
| [GlobalExceptionHandler.java](src/main/java/com/example/abac/common/GlobalExceptionHandler.java) | 예외 → HTTP 상태 매핑. 403(Access/Authorization Denied), 401(Auth), 404(EntityNotFound), 409(IllegalState), 400(Validation/IllegalArg). |

### `config/` — 빈 구성
| 파일 | 역할 |
|---|---|
| [SecurityConfig.java](src/main/java/com/example/abac/config/SecurityConfig.java) | Spring Security 전체 체인. CSRF off, STATELESS 세션, `/auth/**`·`/h2-console/**` permitAll, JWT 필터 삽입, 인증 실패 401. `PasswordEncoder`(BCrypt)·`AuthenticationManager` Bean 등록도 여기. |
| [DataInitializer.java](src/main/java/com/example/abac/config/DataInitializer.java) | 부팅 시 부서 3개(Engineering/Sales/Finance) + 사용자 5명(emp1·emp2·mgr1·fin1·admin1, 비밀번호 `pass`) 시딩. `CommandLineRunner`로 DB 비어있을 때만 동작. |

| [AuthController.java](src/main/java/com/example/abac/security/AuthController.java) | `POST /auth/login` 엔드포인트. |

| [jwt/JwtTokenProvider.java](src/main/java/com/example/abac/security/jwt/JwtTokenProvider.java) | JWT 생성/파싱. claims에 `userId`·`role`·`departmentId` 3개 박음. jjwt 0.12.x. |
| [jwt/JwtAuthenticationFilter.java](src/main/java/com/example/abac/security/jwt/JwtAuthenticationFilter.java) | `OncePerRequestFilter`. `Authorization: Bearer <token>` → `CustomUserPrincipal` 복원 → `SecurityContext` 주입. 실패 시 컨텍스트만 비우고 체인 통과(뒤에서 `anyRequest().authenticated()`가 401 처리). |

| [dto/LoginResponse.java](src/main/java/com/example/abac/security/dto/LoginResponse.java) | `{token, username, role, departmentId}` record. |

| [ExpenseController.java](src/main/java/com/example/abac/domain/expense/ExpenseController.java) | REST 엔드포인트. **권한 매핑이 한눈에 보이는 곳** — 각 메서드의 `@PreAuthorize`가 RBAC/ABAC 규칙을 명시. |

| [dto/UpdateExpenseRequest.java](src/main/java/com/example/abac/domain/expense/dto/UpdateExpenseRequest.java) | Create와 동일 필드. 의미 구분 위해 별도 타입. |
| [dto/ExpenseResponse.java](src/main/java/com/example/abac/domain/expense/dto/ExpenseResponse.java) | `Expense` → 응답 변환. `from(Expense)` 정적 팩토리. |

### `policy/` — ABAC 정책 Bean
| 파일 | 역할 |
|---|---|
| [ExpensePolicy.java](src/main/java/com/example/abac/policy/ExpensePolicy.java) | `@Component("expensePolicy")`. **액션 권한 규칙의 단일 진실 소스**. `canEditDraft` / `canSubmit` / `canApprove` / `canReject` / `canPay` 메서드를 `@PreAuthorize` SpEL에서 `@expensePolicy.canApprove(#id, authentication)` 형태로 호출. 상수 `MANAGER_APPROVAL_LIMIT = 1,000,000`. |

## 요청 한 번의 여정

`POST /expenses/1/approve` (mgr1 토큰) 요청이 들어왔을 때 각 파일이 어떻게 협력하는지:

```
1. HTTP Request [Authorization: Bearer <jwt>]
           │
           ▼
2. SecurityConfig → Filter Chain 구성
           │
           ▼
3. JwtAuthenticationFilter
   - JwtTokenProvider.parse(token) → CustomUserPrincipal 복원
   - UsernamePasswordAuthenticationToken 생성 → SecurityContext 주입
           │
           ▼
4. AuthorizationFilter (anyRequest().authenticated() 통과)
           │
           ▼
5. ExpenseController#approve
   @PreAuthorize("@expensePolicy.canApprove(#id, authentication)")
   │
   │   ◆ SpEL 평가:
   │     ExpensePolicy.canApprove(1L, auth)
   │     ├─ principal.role == MANAGER?
   │     ├─ expense.departmentId == principal.departmentId?
   │     ├─ expense.amount ≤ 1,000,000?
   │     └─ expense.status == SUBMITTED?
   │     → true 이면 메서드 진입, false면 403 (GlobalExceptionHandler)
   │
           ▼
6. ExpenseService#approve
   - requireById(id) → 없으면 404
   - assertStatus(SUBMITTED, APPROVED) → 불일치면 409
   - expense.changeStatus(APPROVED)
           │
           ▼
7. Hibernate flush → DB UPDATE
           │
           ▼
8. ExpenseResponse JSON 반환
```

조회(`GET /expenses`) 플로우는 6번에서 `ExpenseSpecifications.visibleTo(principal)`이 WHERE 절을 주입하는 것이 차이점.

## 두 권한 축의 역할 분담 (핵심 요약)

| 관심사 | 질문 | 담당 | 위치 |
|---|---|---|---|
| **액션 권한** | "이 요청을 수행해도 되는가?" | `@PreAuthorize` + `ExpensePolicy` | Controller 메서드 + [ExpensePolicy.java](src/main/java/com/example/abac/policy/ExpensePolicy.java) |
| **데이터 스코프** | "이 요청이 볼 수 있는 행은?" | JPA `Specification` | [ExpenseSpecifications.java](src/main/java/com/example/abac/domain/expense/ExpenseSpecifications.java) |
| **상태 전이** | "이 상태 변경이 유효한가?" | Service 가드 (`assertStatus`) | [ExpenseService.java](src/main/java/com/example/abac/domain/expense/ExpenseService.java) |
| **인증** | "너 누구?" | JWT 필터 | [JwtAuthenticationFilter.java](src/main/java/com/example/abac/security/jwt/JwtAuthenticationFilter.java) |

이 경계를 흐리지 않는 것이 코드를 깔끔하게 유지하는 핵심이다. 예: 컨트롤러에서 수동으로 소유권 검사하지 말고 Policy에 위임, 서비스에서 역할 분기로 필터링하지 말고 Specification에 위임.
